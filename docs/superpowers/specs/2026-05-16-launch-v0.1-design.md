# VCM v0.1 Launch — Design Spec

**Status:** Approved 2026-05-16
**Sub-skill used:** superpowers:brainstorming → superpowers:writing-plans
**Implementation plan:** [`docs/superpowers/plans/2026-05-16-launch-v0.1.md`](../plans/2026-05-16-launch-v0.1.md)

## Purpose

Convert VCM from a developer-installed CLI (`vcm` via `cargo install` / Nix) into a proper signed, notarized macOS application distributed via DMG, with the first official GitHub release. Reorganize logging to be visible to the host OS (Console.app / `log` CLI), and introduce platform-abstraction trait boundaries so future Linux/Windows work doesn't require a refactor of the core.

## Non-goals

- Sparkle / auto-update (future).
- App Store distribution.
- Linux / Windows implementations of the new platform traits — only macOS impls for v0.1.
- Universal binary (aarch64-only for v0.1 — CoreML is Apple Silicon).
- Homebrew tap (deferred — vcmctl tarball ships separately for that audience).
- Pushing the v0.1.0 tag (user action; CI is set up to react to the tag).

## Pre-flight blockers

Before the v0.1.0 tag can produce a release, the user must provide:

1. **Real `assets/icon/AppIcon.icns`** — generated from a 1024×1024 PNG via `scripts/build-icns.sh`. The icon prompt at `docs/plans/icon-prompt.md` is a starting point.

That is the only hard blocker. Apple Developer Program enrollment is **opportunistic** for v0.1 (see decision 9 below).

## Decisions and rationale

### 1. Architecture: aarch64-apple-darwin only

| Option | Picked? | Why |
|---|---|---|
| aarch64-only | ✅ | CoreML acceleration (the entire reason for the project's whisper.cpp choice) is Apple Silicon only. Intel users would run CPU-only Whisper with substantially worse latency. |
| Universal (lipo) | ❌ | Doubles CI time, doubles DMG size, doubles signing surface, and Intel users get a degraded experience anyway. Easier to add an x86_64 build later if there is documented demand. |

### 2. Minimum macOS: 13.0 Ventura

| Option | Picked? | Why |
|---|---|---|
| 13.0 Ventura | ✅ | Widest user base. Whisper-CoreML works fine on 13+. Apple still ships security updates for it. |
| 14.0 Sonoma | ❌ | Cuts ~15% of users for no APIs we actually need today. |
| 15.0 Sequoia | ❌ | Cuts ~25-30% of users. No 15-specific API in v0.1. |

### 3. Bundler: cargo-packager

| Option | Picked? | Why |
|---|---|---|
| cargo-packager | ✅ | Actively maintained (Tauri team). Has every macOS config we need: `info_plist_path`, `entitlements`, `signing_identity`, `notarization_credentials`, `frameworks`, `minimum_system_version`, `background_app` (= LSUIElement). Single config file, declarative. |
| cargo-bundle | ❌ | Unmaintained (last release Aug 2022). Known issues with workspaces. |
| Hand-rolled bash | ❌ | User explicitly wants to avoid maintaining ad-hoc tooling. |

### 4. Bundle layout: VS Code pattern (CLI bundled, menubar action symlinks into PATH)

| Component | Lives at | Rationale |
|---|---|---|
| `vcm` (menubar launcher) | `Contents/MacOS/vcm` | Single `CFBundleExecutable` per Apple convention |
| `vcmd` (daemon, spawned as subprocess) | `Contents/Resources/vcmd` | cargo-packager's `resources` field copies into `Contents/Resources/` — sufficient for our needs and works without external_binaries renaming |
| `vcmctl` (control CLI) | `Contents/Resources/bin/vcmctl` | Same place VS Code puts `code`, Sublime puts `subl`, etc. |
| `install-vcmctl-cli.sh` (PATH install helper) | `Contents/Resources/install-vcmctl-cli.sh` | Invoked by the menubar via `osascript ... with administrator privileges` to symlink `/usr/local/bin/vcmctl` (or `~/.local/bin/vcmctl` fallback) |

This matches the dominant pattern for desktop apps that ship a CLI: VS Code, Sublime Text, JetBrains IDEs, Docker Desktop, Tailscale, Atom, GitKraken. ([VS Code: Command Line Interface](https://code.visualstudio.com/docs/configure/command-line))

### 5. CLI side-channel: standalone signed `vcmctl` tarball in each release

Even with the in-bundle install action, ship `vcmctl-VERSION-aarch64-apple-darwin.tar.gz` alongside the DMG for users who want just the CLI without installing the .app. ~10 lines of additional CI. Forward-compatible if/when a Homebrew tap is added later.

### 6. Bundle identifier: `com.github.palicand.vcm`

Reverse-DNS of the GitHub repo namespace. Used everywhere consistently: Info.plist `CFBundleIdentifier`, oslog subsystem string, LaunchAgent label. Not `com.palicka.vcm` (the prior aborted attempt used this — explicitly rejected by the user).

### 7. Logging: `tracing-oslog` default, file appender opt-in via `VCM_LOG_FILE=1`

| Option | Picked? | Why |
|---|---|---|
| tracing-oslog default + file opt-in | ✅ | Logs are visible in Console.app and via `log show --predicate 'subsystem == "com.github.palicand.vcm"'` — the native macOS expectation. Dev/troubleshooting users keep the rolling file path. |
| File appender default + oslog opt-in | ❌ | Less macOS-native UX. The whole point of the .app reorg is being a good native citizen. |
| Both always on | ❌ | More IO, more noise. The minority of users who need files can set the env var. |

Subsystem string: `com.github.palicand.vcm`. Categories: `daemon` (from daemon process) and `menubar` (from menubar process).

### 8. Platform abstraction: three traits, macOS impls only

Trait surface kept minimal — only abstract what is actually platform-specific in code today (per YAGNI), plus one trait the user is near-certain to need in v0.2 (Autostart).

| Trait | Method(s) | macOS impl |
|---|---|---|
| `FrontmostApp` | `current() -> Result<String>` | AppleScript via `osascript` (moved from `daemon/src/inject.rs:70`) |
| `LoggingSink` | `layered(registry, opts) -> Subscriber` | `tracing-oslog` + optional rolling file |
| `Autostart` | `enable() / disable() / is_enabled()` | `~/Library/LaunchAgents/com.github.palicand.vcm.plist` round-trip |

Lives in a new `vcm-platform` workspace crate. No menubar UI hook for Autostart in v0.1 — trait is in place so a future "Start at login" menubar item is mechanical.

**Explicitly NOT abstracted:**
- Keystroke injection — already abstracted by `enigo`.
- App data dirs — already abstracted by the `dirs` crate (good enough until Linux/Windows; macOS uses XDG paths today by convention).
- Mic permission prompt — triggered implicitly by first audio access on macOS. Linux/Windows have different models that don't share a usable abstraction.
- Tray icon, Dock hide, RunLoop poke — already abstracted by `tao` and `tray-icon` (cross-platform crates that handle the macOS quirks internally).
- Bundle assembly — separate concern; lives in `Packager.toml`.

### 9. Release pipeline: manual tag push → CI → DRAFT release (signing/notarization OPPORTUNISTIC)

**Rationale for the opportunistic stance:** the user is starting on a free Apple Developer account. A Developer ID Application certificate and notarization both require a **paid** Apple Developer Program enrollment ($99/year) — free accounts cannot generate either. Rather than block v0.1.0 on that recurring cost, ship ad-hoc-signed DMGs for v0.1 (a well-trodden pattern for FOSS Mac apps: VS Codium, MacPass, MeetingBar in its early days). End users hit a single Gatekeeper warning on first launch which they bypass by right-clicking → Open OR running `xattr -d com.apple.quarantine /Applications/VCM.app`. The release workflow is written so the only thing required to flip to fully-signed-and-notarized later is adding the six secrets — no code change.

Triggered by pushing a `v*` tag. The release workflow on `macos-14`:

1. Fails fast if `assets/icon/AppIcon.icns` is missing.
2. Builds `cargo packager --release --target aarch64-apple-darwin` (produces only the `.app`; DMG is built later via `hdiutil` after the `.app` is signed — because cargo-packager seals the DMG before any post-hoc signing is possible).
3. **If signing secrets are present** (`APPLE_DEVELOPER_ID`, `APPLE_CERT_P12_BASE64`, `APPLE_CERT_PASSWORD`): codesigns the .app with hardened runtime + entitlements + Developer ID + timestamp; signs the DMG. **Otherwise**: ad-hoc signs (`codesign -s -`) the .app and skips DMG signing.
4. **If notarization secrets are present** (`APPLE_API_KEY_BASE64`, `APPLE_API_KEY_ID`, `APPLE_API_KEY_ISSUER_ID`): runs `xcrun notarytool submit --wait` then `stapler staple` + `stapler validate`. **Otherwise**: skips with a warning in the job log.
5. Builds the standalone vcmctl tarball; signs it if signing secrets are present, ad-hoc signs otherwise.
6. Generates SHA256 checksums for all artifacts.
7. Publishes a **draft** release with all artifacts. User reviews + clicks "Publish".

Draft (not auto-publish) gates against accidental public releases from typoed tags or untested builds.

**Six secrets** to add later (when paid enrollment happens):
- `APPLE_DEVELOPER_ID` — e.g., `Developer ID Application: Andrej Palicka (TEAMID)` (from Keychain or Apple Developer portal)
- `APPLE_CERT_P12_BASE64` — Developer ID cert exported from Keychain, base64-encoded
- `APPLE_CERT_PASSWORD` — .p12 export password
- `APPLE_API_KEY_BASE64` — App Store Connect Team API key (.p8) with Developer role, base64-encoded. From [App Store Connect → Users and Access → Integrations → Team Keys](https://appstoreconnect.apple.com/access/api).
- `APPLE_API_KEY_ID` — 10-char key ID shown after creating the key
- `APPLE_API_KEY_ISSUER_ID` — UUID shown above the "Active" table on the same page

### 9b. Gatekeeper bypass for v0.1 end users

While VCM ships ad-hoc-signed, README must include a "First launch" section:

> On first launch, macOS may show "VCM cannot be opened because the developer cannot be verified." This is because VCM v0.1 ships without an Apple-issued signature.
>
> Bypass: right-click `VCM.app` in Finder → **Open** → "Open" in the confirmation dialog. This only needs to be done once.
>
> Alternative: `xattr -d com.apple.quarantine /Applications/VCM.app`.

This text moves into a `## First launch` section of the README and is also surfaced in the GitHub release notes for v0.1.

### 10. cargo install / Nix flake compatibility

The DMG + tarball release artifacts do not affect the `cargo install --path .` or Nix flake install paths — they consume source from the tag, not the release artifacts. No regression for existing users.

## Out of scope (do not implement)

- Sparkle / auto-update.
- Linux / Windows implementations of platform traits.
- App Store distribution.
- Homebrew tap (deferred).
- Pushing the `v0.1.0` tag (user action).
- Menubar UI hook for the `Autostart` trait (trait exists; UI is v0.2 work).
